⚗ Feynman — What is a Database?

Imagine you're running a school with 500 students. You could store everything in Excel sheets — but if you want "all students who scored above 80 in Python AND are in the CS department", you'd have to do that manually. A database is an organised collection of data, and SQL (Structured Query Language) is the language you use to talk to it. You ask questions like "give me all students from Chennai" and the database answers instantly — even if there are 10 million students. SQL is not a programming language — it's a query language. You describe what you want, not how to get it.

🗣 Socratic Dialogue — Before Writing a Single Query

S

A list works for 10 students. What if I need everyone who scored above 80 in exactly 3 subjects and is in the 3rd year? How fast can a Python list do that?

T
Right. A database is built specifically for that — it can answer that question across millions of rows in milliseconds. That's why every web app — Instagram, YouTube, your college portal — runs on a database. Now — what do you think a "table" in a database looks like?
S
Like a spreadsheet? Rows and columns?
T
Exactly. Each table is like one Excel sheet. Each row is one record. Each column is one attribute. And SQL is how you read, write, update, and delete those records.

DDL — Building the Structure
ddl_create.sql
-- ───────────────────────────────────────────────────── -- First, create the database itself (a container for tables) -- ───────────────────────────────────────────────────── CREATE DATABASE school_db; -- ① Create a new database called school_db USE school_db; -- ② Switch to working inside school_db -- Create the students table with column definitions CREATE TABLE students ( -- ③ Create a table named students id INT AUTO_INCREMENT PRIMARY KEY, -- ④ Unique ID, auto-counts up, main identifier name VARCHAR(100) NOT NULL, -- ⑤ Text up to 100 chars, cannot be empty age INT, -- ⑥ Whole number (no decimal) department VARCHAR(50), -- ⑦ Text up to 50 chars marks DECIMAL(5,2), -- ⑧ Number with 5 digits, 2 decimal places city VARCHAR(50) DEFAULT 'Chennai', -- ⑨ Optional: default value if not given enrolled_on DATE -- ⑩ Date format: YYYY-MM-DD ); -- Modify an existing table (add a new column) ALTER TABLE students ADD COLUMN email VARCHAR(150) UNIQUE; -- ⑪ Add email column, no two students can share it -- Remove a column you no longer need ALTER TABLE students DROP COLUMN city; -- ⑫ Delete the city column entirely -- Check the current structure of the table DESCRIBE students; -- ⑬ Shows all columns, types, constraints -- Permanently delete the table (no undo!) -- DROP TABLE students; -- ⑭ Careful — destroys all data forever
// Line-by-Line Explanation
①
CREATE DATABASE school_db — Creates the database container. Think of it as creating a new folder. All related tables will live inside this database.
②
USE school_db — Tells MySQL "all my next commands should run inside this database." Without this, you'd get errors because MySQL doesn't know which database to look in.
③
CREATE TABLE students ( ... ) — Creates a new table. Everything inside the parentheses defines the columns. Each line inside is one column with its data type and rules.
④
INT AUTO_INCREMENT PRIMARY KEY — Three things together: INT = whole number, AUTO_INCREMENT = MySQL automatically gives 1, 2, 3... so you never set this manually, PRIMARY KEY = this is the unique identifier for every row.
⑤
VARCHAR(100) NOT NULL — VARCHAR = variable-length text (only uses as much space as needed), (100) = maximum 100 characters, NOT NULL = this field cannot be left empty — MySQL will reject the row if name is missing.
⑧
DECIMAL(5,2) — Stores a precise decimal number. (5,2) means: 5 total digits, 2 after the decimal point. So: 999.99 is valid. Use DECIMAL for money/marks — never FLOAT (FLOAT has rounding errors).
⑨
DEFAULT 'Chennai' — If a new student is added without specifying a city, MySQL automatically fills in 'Chennai'. Saves repetitive data entry.
DML — Working With Data
dml_crud.sql
-- ══════════════════════════════════════════════ -- INSERT — Adding new rows into the table -- ══════════════════════════════════════════════ -- Insert ONE student (providing all columns) INSERT INTO students (name, age, department, marks) VALUES ('Priya', 20, 'CS', 88.50); -- id is AUTO_INCREMENT so we skip it — MySQL sets it automatically -- Insert MULTIPLE students at once (more efficient than one-by-one) INSERT INTO students (name, age, department, marks) VALUES ('Ravi', 21, 'CS', 75.00), -- first student ('Meena', 20, 'ECE', 92.00), -- second student ('Arjun', 22, 'MECH', 65.50), -- third student ('Kavi', 20, 'CS', 55.00); -- fourth student -- ══════════════════════════════════════════════ -- UPDATE — Changing existing row data -- ══════════════════════════════════════════════ -- Update ONE specific student (use WHERE to target exactly) UPDATE students SET marks = 90.00 -- change marks to 90.00 WHERE id = 1; -- ONLY for the row where id = 1 (Priya) -- Update MULTIPLE columns at once for a student UPDATE students SET marks = 78.00, department = 'IT' -- comma-separated to update multiple columns WHERE name = 'Ravi'; -- target by name instead of id -- ══════════════════════════════════════════════ -- DELETE — Removing rows -- ══════════════════════════════════════════════ -- Delete ONE specific student DELETE FROM students WHERE id = 5; -- remove the row with id = 5 only -- Delete all students from a department DELETE FROM students WHERE department = 'MECH'; -- removes ALL MECH students
Always use WHERE with UPDATE and DELETE. Running DELETE FROM students; without a WHERE clause deletes every single row in the table instantly. This is the most dangerous SQL mistake beginners make. Always double-check your WHERE condition before executing.
DQL — Querying Data (The Real Power)
dql_select.sql
-- ── SELECT BASICS ──────────────────────────────────── SELECT * FROM students; -- * means ALL columns. Shows every column and every row. Use for exploration, not production. SELECT name, marks FROM students; -- Show ONLY name and marks columns. Always prefer specific columns over * SELECT DISTINCT department FROM students; -- DISTINCT removes duplicates. Shows each department name only once. -- ── WHERE — Filtering rows ─────────────────────────── SELECT * FROM students WHERE marks > 75; -- Only rows where marks column value is greater than 75 SELECT * FROM students WHERE department = 'CS' AND marks > 80; -- AND: BOTH conditions must be true. CS students who also scored above 80. SELECT * FROM students WHERE department = 'CS' OR department = 'IT'; -- OR: Either condition is true. Students in CS or IT. SELECT * FROM students WHERE department IN ('CS', 'IT', 'ECE'); -- IN is cleaner than multiple OR conditions. Same result, more readable. SELECT * FROM students WHERE marks BETWEEN 60 AND 90; -- BETWEEN is inclusive — includes 60 and 90. Same as marks >= 60 AND marks <= 90. SELECT * FROM students WHERE name LIKE 'P%'; -- LIKE with % wildcard: names that START with P. % = "any characters after" SELECT * FROM students WHERE email IS NULL; -- IS NULL: rows where email column has no value. Cannot use = NULL — use IS NULL. -- ── ORDER BY — Sorting results ─────────────────────── SELECT name, marks FROM students ORDER BY marks DESC; -- DESC = highest to lowest (descending). ASC = lowest to highest (ascending, the default). SELECT name, department, marks FROM students ORDER BY department ASC, marks DESC; -- Sort by department (A-Z first), then within each department sort by marks (highest first) SELECT name, marks FROM students ORDER BY marks DESC LIMIT 3; -- LIMIT 3: show only the top 3 results. Used for "top N" queries and pagination. -- ── AGGREGATE FUNCTIONS ────────────────────────────── SELECT COUNT(*) AS total_students, -- count all rows AVG(marks) AS avg_marks, -- average of marks column MAX(marks) AS highest, -- highest marks value MIN(marks) AS lowest, -- lowest marks value SUM(marks) AS total_marks -- sum of all marks FROM students; -- AS gives the result column a readable name (alias) -- ── GROUP BY — Aggregate per group ─────────────────── -- How many students are in each department? SELECT department, COUNT(*) AS student_count, AVG(marks) AS avg_marks FROM students GROUP BY department; -- GROUP BY groups all CS students together, all ECE together, etc., -- then COUNT(*) and AVG() run on each group separately -- HAVING — filter AFTER grouping (WHERE filters before grouping) SELECT department, AVG(marks) AS avg_marks FROM students GROUP BY department HAVING AVG(marks) > 70; -- HAVING filters groups. Here: only show departments where average marks > 70. -- WHERE cannot use aggregate functions. HAVING can.
// Key Concept Breakdown
LIKE
'P%' = starts with P, '%a' = ends with a, '%ri%' = contains "ri" anywhere. The % is a wildcard for zero or more characters. The _ underscore wildcard means exactly one character.
NULL
NULL means no value — not 0, not empty string, but literally "unknown/missing." You cannot compare with = NULL. Always use IS NULL or IS NOT NULL.
GROUP BY
Used with aggregate functions (COUNT, AVG, SUM, MAX, MIN). Every column in SELECT that is NOT inside an aggregate function MUST be in GROUP BY. Think: "group the rows, then count/average within each group."
WHERE vs HAVING
WHERE filters individual rows before grouping. HAVING filters groups after grouping. Rule: if you need to use COUNT(), AVG() etc. in a filter — use HAVING.

Q1
What is the difference between WHERE and HAVING? Why can't you use WHERE AVG(marks) > 70?
›
WHERE filters individual rows before any grouping happens — it runs on raw data. HAVING filters after GROUP BY creates groups. AVG(marks) is an aggregate — it doesn't exist as a value for individual rows. It only exists after rows are grouped. So WHERE literally can't see it yet.
Q2
What happens if you run UPDATE students SET marks = 100 without a WHERE clause?
›
Every single student in the table gets their marks set to 100. There is no automatic undo in MySQL unless you're inside a transaction. This is one of the most catastrophic beginner mistakes in production databases. Always write WHERE first, then the rest of the UPDATE.
Q3
What is the difference between DELETE FROM students and DROP TABLE students?
›
DELETE FROM students removes all rows but keeps the table structure (columns, constraints) intact. DROP TABLE students destroys the entire table — structure, data, everything — completely gone. TRUNCATE TABLE is like DELETE but faster and non-transactional.

⚗ Feynman — Why Joins Exist

Good database design stores each type of data in its own table. Student details in a students table. Course details in a courses table. Grades in an enrollments table. This avoids repeating the same data everywhere. But now, when you want "show me each student's name along with the courses they're enrolled in and their grades," you need to combine these tables. That's what a JOIN does — it stitches tables together on a matching column, like connecting two puzzle pieces that share an edge.

🗣 Socratic Dialogue
T
If I have a students table and a courses table, how do I know which student is enrolled in which course?
S
Store the student ID in the courses table?
T
Exactly — that's called a foreign key. The enrollment table holds a student_id and a course_id — it links the two. Now, how do I get the student's actual name alongside the course name in one result?
S
You'd have to join the tables on the matching IDs?
T
Perfect. That's a JOIN. The matching column is the bridge between the two tables. Now — what if some students haven't enrolled in any course yet — would an INNER JOIN show them?
S
No? Because there's no matching row in enrollments?
T
Right — INNER JOIN only shows rows that have a match in BOTH tables. To include students with no enrollment, you'd use a LEFT JOIN — it keeps all rows from the left (students) table.
The Four Join Types
INNER JOIN
🟦🟩
Returns ONLY rows where a match exists in BOTH tables. Non-matching rows from either side are excluded.
Students who ARE enrolled in a course
LEFT JOIN (LEFT OUTER)
🟦⬜
Returns ALL rows from the LEFT table. Non-matching rows from the right table show as NULL.
ALL students — enrolled or not
RIGHT JOIN (RIGHT OUTER)
⬜🟩
Returns ALL rows from the RIGHT table. Non-matching rows from the left table show as NULL.
ALL courses — enrolled or not
FULL OUTER JOIN
🟦🟣🟩
Returns ALL rows from BOTH tables. Non-matching rows on either side show as NULL. (MySQL: simulate with UNION)
All students and all courses, matched or not
joins.sql — Setup + All Join Types
-- First: the two tables we'll join CREATE TABLE courses ( id INT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(100), credits INT ); CREATE TABLE enrollments ( id INT AUTO_INCREMENT PRIMARY KEY, student_id INT, -- links to students.id course_id INT, -- links to courses.id grade CHAR(2), FOREIGN KEY (student_id) REFERENCES students(id), -- enforces relationship FOREIGN KEY (course_id) REFERENCES courses(id) -- cannot add invalid IDs ); -- ── INNER JOIN ──────────────────────────────────────── -- Only students who are actually enrolled in a course SELECT s.name AS student_name, -- s. prefix = from students table c.name AS course_name, -- c. prefix = from courses table e.grade -- e. prefix = from enrollments table FROM students AS s -- alias 's' for shorter typing INNER JOIN enrollments AS e ON s.id = e.student_id -- match students to enrollments INNER JOIN courses AS c ON e.course_id = c.id; -- match enrollments to courses -- ── LEFT JOIN ───────────────────────────────────────── -- ALL students — those without enrollment show NULL for course columns SELECT s.name AS student, c.name AS course, -- NULL if student not enrolled e.grade -- NULL if student not enrolled FROM students AS s LEFT JOIN enrollments AS e ON s.id = e.student_id LEFT JOIN courses AS c ON e.course_id = c.id; -- Find students who haven't enrolled in ANYTHING SELECT s.name FROM students AS s LEFT JOIN enrollments AS e ON s.id = e.student_id WHERE e.student_id IS NULL; -- NULL means no match = not enrolled
Subqueries — A Query Inside a Query
subqueries.sql
-- A subquery is a SELECT inside another SELECT (or WHERE, FROM, HAVING) -- MySQL runs the inner query first, then uses its result in the outer query -- ── Example 1: Students who scored above the class average ── SELECT name, marks FROM students WHERE marks > ( SELECT AVG(marks) FROM students -- inner query: calculate average first ); -- Step 1: Inner query runs → AVG(marks) = 75.0 -- Step 2: Outer query runs → WHERE marks > 75.0 -- ── Example 2: Students enrolled in more than 2 courses ── SELECT name FROM students WHERE id IN ( SELECT student_id -- inner query: find IDs of busy students FROM enrollments GROUP BY student_id HAVING COUNT(*) > 2 -- only IDs with more than 2 course rows ); -- The outer query receives a LIST of IDs and finds those student names
Views — Saved Queries You Can Reuse
A View is like a named shortcut. Instead of writing a complex 5-table JOIN every time you need the same report, you create a view once and query it like a regular table. The view doesn't store data — every time you query it, it runs the underlying query fresh.
views.sql
-- Create a view: saves the JOIN query with a name CREATE VIEW student_courses AS -- ① give this view a name SELECT s.name AS student, c.name AS course, e.grade FROM students AS s INNER JOIN enrollments AS e ON s.id = e.student_id INNER JOIN courses AS c ON e.course_id = c.id; -- Now query it EXACTLY like a table — no JOIN needed! SELECT * FROM student_courses; -- ② use view like a table SELECT * FROM student_courses WHERE grade = 'A'; -- filter on top of view -- Modify a view later CREATE OR REPLACE VIEW student_courses AS -- ③ update the view definition SELECT s.name, c.name AS course, e.grade, s.marks FROM students s JOIN enrollments e ON s.id = e.student_id JOIN courses c ON e.course_id = c.id; -- Remove a view DROP VIEW student_courses; -- ④ deletes the view, NOT the data
Stored Procedures — Reusable SQL Logic
⚗ Feynman — Stored Procedures

A stored procedure is like a function inside the database. You write SQL logic once, save it with a name, and call it with parameters anytime you need it. Instead of your Python app sending a long SQL query every time, it sends one short command: CALL get_toppers(80). The logic lives in the database — closer to the data. This reduces network traffic and centralises business logic.

stored_procedures.sql
-- Change the delimiter so MySQL doesn't confuse ; inside the procedure DELIMITER // -- ① change command-end symbol from ; to // CREATE PROCEDURE get_toppers(IN min_marks DECIMAL(5,2)) -- IN: parameter going IN to the procedure. Name = min_marks, Type = DECIMAL BEGIN -- ② marks the start of the procedure body SELECT name, department, marks FROM students WHERE marks >= min_marks -- ③ use the parameter in the query ORDER BY marks DESC; END // -- ④ marks the end of the procedure body DELIMITER ; -- ⑤ restore the delimiter back to normal -- Call the procedure — like calling a function CALL get_toppers(80.00); -- ⑥ runs the query with min_marks = 80.00 CALL get_toppers(60.00); -- ⑦ same logic, different threshold -- ── Procedure with OUT parameter (returns a value) ── DELIMITER // CREATE PROCEDURE count_dept(IN dept_name VARCHAR(50), OUT total INT) BEGIN SELECT COUNT(*) INTO total -- INTO stores result INTO the OUT variable FROM students WHERE department = dept_name; END // DELIMITER ; -- Call with an output variable CALL count_dept('CS', @cs_count); -- @variable = session variable SELECT @cs_count; -- retrieve the output value -- Remove a procedure DROP PROCEDURE IF EXISTS get_toppers;

Q1
What is the difference between INNER JOIN and LEFT JOIN? When would you use each?
›
INNER JOIN: Only returns rows with a match in BOTH tables. Use when you only care about records that have corresponding data on both sides (e.g. students who are enrolled). LEFT JOIN: Returns ALL rows from the left table + matching rows from the right. Rows with no match get NULL. Use when you want all records from the primary table regardless of whether they have related data (e.g. all students including unenrolled ones).
Q2
What is a View and how is it different from a table?
›
A View is a saved query — it's a virtual table. It stores the query definition, not the actual data. Every time you query a view, it re-runs the underlying query. A real table stores physical data on disk. Views are useful for: (1) simplifying complex queries, (2) restricting access to certain columns, (3) giving a stable interface even if the underlying tables change.

⚗ Feynman — How Python Talks to MySQL

Python and MySQL are two separate programs running on your computer. To make them talk, you need a connector — a bridge. The mysql-connector-python library is that bridge. Here's the flow: Python creates a Connection (like opening a phone call to MySQL), then creates a Cursor (like a pen that can write and read), uses the cursor to execute SQL, and then either commits (saves changes) or fetches (reads results). When done, both the cursor and connection are closed to free resources.

import mysql.connector from mysql.connector import Error # ══════════════════════════════════════════════════════ # CREATE — Insert a new student into the database # ══════════════════════════════════════════════════════ def add_student(name, age, department, marks): conn = mysql.connector.connect(host="localhost", user="root", password="password", database="school_db") cursor = conn.cursor() # ① Create a cursor — it's the "pen" that executes SQL query = """ INSERT INTO students (name, age, department, marks) VALUES (%s, %s, %s, %s) """ # ② %s are placeholders — MySQL replaces them safely with the actual values # NEVER put Python variables directly into the query string (SQL injection risk!) values = (name, age, department, marks) # ③ A tuple of actual values — order must match %s cursor.execute(query, values) # ④ Send query + values to MySQL — MySQL fills %s safely conn.commit() # ⑤ CRITICAL: saves the change permanently. Without this, INSERT is lost! print(f"✅ Student added with ID: {cursor.lastrowid}") # ⑥ lastrowid = the auto-generated id cursor.close() # ⑦ Close cursor first (good practice) conn.close() # ⑧ Close connection — frees MySQL server resources # ══════════════════════════════════════════════════════ # READ — Fetch students from the database # ══════════════════════════════════════════════════════ def get_all_students(): conn = mysql.connector.connect(host="localhost", user="root", password="password", database="school_db") cursor = conn.cursor() cursor.execute("SELECT id, name, department, marks FROM students") # No values tuple needed for SELECT without parameters rows = cursor.fetchall() # ⑨ Fetch ALL result rows — returns list of tuples # fetchone() → one row (or None if no more rows) # fetchmany(5) → next 5 rows as list # fetchall() → ALL remaining rows as list for row in rows: # ⑩ Each row is a tuple: (id, name, dept, marks) print(f"ID:{row[0]} | {row[1]} | {row[2]} | {row[3]}") # row[0]=id, row[1]=name, row[2]=department, row[3]=marks cursor.close(); conn.close() return rows # ⑪ Return the list so the caller can use it # ── Better: use DictCursor to get rows as dicts instead of tuples def get_students_as_dicts(): conn = mysql.connector.connect(host="localhost", user="root", password="password", database="school_db") cursor = conn.cursor(dictionary=True) # ⑫ dictionary=True → rows as dicts not tuples cursor.execute("SELECT * FROM students") rows = cursor.fetchall() for row in rows: print(f"Name: {row['name']}, Marks: {row['marks']}") # row['name'] is much clearer than row[1]! cursor.close(); conn.close() # ══════════════════════════════════════════════════════ # UPDATE — Modify existing student data # ══════════════════════════════════════════════════════ def update_marks(student_id, new_marks): conn = mysql.connector.connect(host="localhost", user="root", password="password", database="school_db") cursor = conn.cursor() query = "UPDATE students SET marks = %s WHERE id = %s" # Two %s placeholders: first for new_marks, second for student_id cursor.execute(query, (new_marks, student_id)) # ⑬ tuple order must match %s order conn.commit() # ⑭ Must commit UPDATE too — or change is lost print(f"✅ Updated {cursor.rowcount} row(s)") # ⑮ rowcount = how many rows were affected cursor.close(); conn.close() # ══════════════════════════════════════════════════════ # DELETE — Remove a student # ══════════════════════════════════════════════════════ def delete_student(student_id): conn = mysql.connector.connect(host="localhost", user="root", password="password", database="school_db") cursor = conn.cursor() cursor.execute("DELETE FROM students WHERE id = %s", (student_id,)) # Note the comma: (student_id,) is a one-item tuple, NOT just parentheses conn.commit() print(f"🗑️ Deleted student with ID {student_id}") cursor.close(); conn.close() # ══════════════════════════════════════════════════════ # Running everything together # ══════════════════════════════════════════════════════ if __name__ == "__main__": add_student("Priya", 20, "CS", 88.50) # Insert get_all_students() # Read all update_marks(1, 92.00) # Update id=1 get_all_students() # Verify update
// Critical Concepts
⑤ commit()
The most forgotten step. INSERT, UPDATE, DELETE are not saved to disk until you call conn.commit(). MySQL keeps the changes in a temporary transaction. If the program crashes before commit(), everything is rolled back automatically. For SELECT (read-only), no commit is needed.
② %s
The %s placeholder is how you safely pass values. MySQL's driver replaces them after properly escaping special characters. Never do: query = f"INSERT ... VALUES ('{name}')" — this is SQL injection vulnerable.
⑨ fetchall()
Returns a list of tuples. Each tuple is one row. Each element in the tuple corresponds to one column in the order you selected them. For easier access, use dictionary=True cursor to get row['name'] instead of row[1].
⑮ rowcount
cursor.rowcount tells you how many rows were affected by the last INSERT/UPDATE/DELETE. If you UPDATE with a WHERE that matches nothing, rowcount will be 0 — a useful way to detect "nothing changed."
SQL Injection — The Most Important Security Concept
SQL Injection is the most common web security vulnerability. If you build a SQL query by directly inserting user input, a malicious user can type SQL code as their "name" and destroy your database or steal all data. Parameterized queries (%s placeholders) completely prevent this.
sql_injection.py — Dangerous vs Safe

Q1
What is a cursor and why do you need it if you already have a connection?
›
A connection is the open channel to MySQL — like a phone line. A cursor is the actual tool that sends queries and reads results through that channel — like the person speaking on the phone. One connection can have multiple cursors. The cursor maintains the state of your query — which row you're currently reading, how many rows were affected, etc.
Q2
Why is conn.commit() required after INSERT but not after SELECT?
›
MySQL wraps write operations (INSERT, UPDATE, DELETE) in a transaction by default. The transaction is a "draft" — it's not permanent until committed. commit() finalises the draft. SELECT is a read-only operation — it doesn't change any data, so there's nothing to commit. Without commit, your inserts appear to work (no error) but disappear when the connection closes.
Q3
Explain SQL injection with a real example and explain why %s prevents it.
›
If a user types ' OR '1'='1 as their name, a string-formatted query becomes WHERE name = '' OR '1'='1' — which is always true — leaking all rows. With %s, MySQL receives the value separately from the query structure. It escapes special characters before inserting the value. The single quote in the user's input gets escaped to \' — treated as literal text, not SQL syntax. The query structure is fixed; the value cannot alter it.

Introduction to SQLite
Python's built-in database — no server, no installation, no configuration. A single file IS the database. Perfect for development, testing, and small applications.
sqlite3 module serverless:memory: file-based DBDjango default
⚗ Feynman — What is SQLite?

MySQL is a full server — it runs as a separate process, requires installation, has users and passwords, and handles multiple simultaneous connections. SQLite is the opposite: the entire database lives in a single file on your hard disk. No server process. No network. No login. Python's sqlite3 module is built right in — you don't install anything. You import it and start querying. The SQL syntax is almost identical to MySQL. This is why Django uses SQLite by default — developers can start immediately without setting up a server. When you're ready to deploy, you switch to MySQL or PostgreSQL.

// Key SQLite vs MySQL Differences
? vs %s
SQLite uses ? as its placeholder. MySQL uses %s. Both prevent SQL injection in exactly the same way. When you switch from SQLite to MySQL in Django, this is handled automatically — you never see it.
:memory:
sqlite3.connect(":memory:") creates the entire database in RAM. Nothing is written to disk. The database is completely destroyed when the program ends. Perfect for automated tests — no cleanup needed, no leftover test files.
row_factory
Setting conn.row_factory = sqlite3.Row makes rows accessible by column name instead of index. It's the SQLite equivalent of MySQL's dictionary=True cursor.
executemany
cursor.executemany(query, list_of_tuples) inserts all rows in one call — much faster than calling execute() in a loop. Internally it uses a single transaction for all rows.

③ Does NOT automatically close the connection (unlike file with-statement) # Best practice: still call conn.close() after the with block cursor = conn.cursor() # All of this is wrapped in one transaction cursor.execute("INSERT INTO students (name, age, department, marks) VALUES (?,?,?,?)", ("Test", 20, "CS", 70)) cursor.execute("UPDATE students SET marks = marks + 5 WHERE department = ?", ("CS",)) # If any line above raises an exception, BOTH changes are rolled back # Either both succeed, or neither does — this is ATOMIC operation results = cursor.execute("SELECT name, marks FROM students").fetchall() # Can chain .fetchall() directly onto .execute() for reads for name, marks in results: # tuple unpacking — clean way to read rows print(f" {name}: {marks}") # 'with' block ends — transaction is committed if no exception occurred # enable column name access return conn def setup(self): """Create tables if they don't exist yet.""" with self._get_conn() as conn: conn.execute(""" CREATE TABLE IF NOT EXISTS students ( id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT NOT NULL, dept TEXT, marks REAL ) """) def add(self, name, dept, marks): with self._get_conn() as conn: conn.execute( "INSERT INTO students (name, dept, marks) VALUES (?,?,?)", (name, dept, marks) ) print(f"Added: {name}") def get_all(self): with self._get_conn() as conn: rows = conn.execute("SELECT * FROM students ORDER BY marks DESC").fetchall() return [dict(row) for row in rows] # Q1
What does sqlite3.connect(":memory:") create? Why is it useful for testing?
›
Creates a completely in-memory database — nothing is written to disk. When the program ends (or the connection closes), the database is gone permanently. For testing: each test function can create a fresh :memory: database, run operations, and assert results — without creating, managing, or deleting test files. Tests are faster (RAM vs disk) and cleaner (no cleanup).
Q2
SQLite uses ? as a placeholder and MySQL uses %s. If you switch databases, do you have to rewrite all your queries?
›
In raw Python code — yes, you'd need to change every ? to %s. This is one reason frameworks like Django ORM are valuable — Django translates your Python code into the correct SQL for whatever database you're using. You write Student.objects.filter(marks__gt=80) once, and Django generates the right SQL for SQLite during development and MySQL in production. The placeholder difference is hidden from you.
Q3
When should a developer choose SQLite over MySQL for a project?
›
Choose SQLite for: single-user apps, mobile apps, desktop tools, prototypes, scripts that process data, automated testing. Choose MySQL for: web apps with multiple users, any app where multiple processes write data simultaneously, production deployments, applications needing fine-grained user permissions. Django's golden rule: develop with SQLite, deploy with PostgreSQL/MySQL.
